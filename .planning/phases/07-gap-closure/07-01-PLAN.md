    The query command already has manifest pre-detection via EXTENSION_MAP (lines 42-60 of query-cmd.ts),
    which was implemented in Phase 5. The audit flagged this as "implicit" because it scans file extensions
    rather than storing indexed types explicitly in the manifest.

    Harden the existing approach with a minor improvement:

    1. **Add early exit when no queryable types detected**: After the typesToQuery array is built (line 60),
       if `typesToQuery.length === 0` AND `options.type` was not explicitly set, emit a NO_INDEX error:
       ```
       if (typesToQuery.length === 0 && !options.type) {
         const { emitError } = await import('../errors.js');
         emitError(
           { code: 'NO_INDEX', message: 'No indexed content found', suggestion: 'Run `ez-search index .` first' },
           options.format === 'text' ? 'text' : 'json'
         );
       }
       ```
       This handles the case where the manifest exists but has zero files (e.g., after `--clear`
       without re-indexing), giving a clear error instead of silently returning empty results.

    2. **Add a comment clarifying the pre-detection strategy** at the top of the auto-detect block
       (around line 47) to document the approach for future maintainers:
       ```
       // Pre-detect indexed types from manifest: only load models for types that have data.
       // This avoids loading Jina when only text is indexed (or Nomic when only code is indexed).
       ```

    Do NOT change the fundamental approach (EXTENSION_MAP scanning). It is correct and efficient.
    Do NOT add manifest metadata fields -- that would be over-engineering for this use case.
  </action>
