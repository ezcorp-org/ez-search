**index-cmd.ts rewrite:**

Remove the early-exit stubs for text/image pipelines (lines 36-43).

Change the default behavior when `--type` is not specified:
- OLD: `typeFilter` defaults to `'code'`
- NEW: When `--type` is not specified, run ALL three pipelines on their respective file types.
- When `--type` IS specified, run only that one pipeline.

Restructure the pipeline into a loop over active types:

```
const typesToIndex: FileType[] = options.type
  ? [options.type as FileType]
  : ['code', 'text', 'image'];
```

For each type in `typesToIndex`:
1. Scan files with `typeFilter` set to that type
2. If no files found for this type, skip
3. Run the appropriate pipeline:

**Code pipeline** (existing logic, extract into helper or inline):
- Load Jina tokenizer + chunker.ts `chunkFile()`
- Embed with `createEmbeddingPipeline('code')`
- Insert into `col768` with `modelId` = jina model ID

**Text pipeline** (new):
- For each text file:
  - If `.pdf` extension: read as Buffer, call `extractPdfText(buffer)` from text-chunker, then `chunkTextFile(text)`
  - Otherwise: read as UTF-8 string, call `chunkTextFile(text)`
- Prefix each chunk text with `"search_document: "` before embedding (Nomic task prefix requirement)
- Embed with `createEmbeddingPipeline('text')`
- Insert into `col768` with `modelId` = nomic model ID
- Use `makeChunkId(relPath, chunkIndex)` for chunk IDs (same format as code)
- Store metadata: `filePath`, `chunkIndex`, `modelId`, `lineStart: 0`, `lineEnd: 0`, `chunkText` (without the prefix)

**Image pipeline** (new):
- For each image file:
  - Call `createImageEmbeddingPipeline()` (load once, reuse)
  - Call `pipeline.embedImage(absolutePath)` for each image
  - Insert into `col512` with metadata: `filePath`, `chunkIndex: 0`, `modelId: 'Xenova/clip-vit-base-patch32'`, `lineStart: 0`, `lineEnd: 0`, `chunkText: ''`
  - Use `makeChunkId(relPath, 0)` for ID (one image = one vector)

**Manifest handling:**
- The manifest tracks ALL types together (same `.ez-search-cache` file)
- Change detection works the same for all types (mtime+size fast path, hash confirmation)
- For images: chunks array has exactly one entry (chunkIndex 0)

**Optimize order:**
- After ALL types are processed: optimize col768, then optimize col512 (only if image files were indexed)
- Save manifest AFTER both optimizations succeed

**Output stats:**
- Aggregate stats across all types (filesIndexed, chunksCreated, etc.)
- In text format output, mention which types were indexed: "indexed 5 code files, 3 text files, 2 images"

**DRY principle:** The code/text pipelines share most logic (scan, manifest check, chunk, batch embed, insert into col768). Extract shared logic into a helper function rather than duplicating the manifest/embed/insert loop. The key differences are:
- Chunker function (chunkFile vs chunkTextFile)
- Model type ('code' vs 'text')
- Task prefix (none for code, "search_document: " for text)
- Tokenizer loading (needed for code only)
- PDF extraction (text only)

Consider a `runEmbeddingPipeline(type, files, col, ...)` helper that both code and text call.

<task type="auto">
  <name>Task 2: Rewrite query-cmd for multi-collection grouped search</name>
  <files>src/cli/commands/query-cmd.ts</files>
  <action>
Rewrite query-cmd.ts to search across all indexed collections:

**Auto-detection logic:**
When `--type` is not specified:
1. Query col-768 with Jina code embedding -> code results
2. Query col-768 with Nomic text embedding (prefix query with `"search_query: "`) -> text results
3. Skip col-512 for text queries (CLIP text encoder not loaded in Phase 5 — research recommendation)
4. Filter code results: only keep those where `metadata.modelId` contains 'jina' (or starts with 'jinaai/')
5. Filter text results: only keep those where `metadata.modelId` contains 'nomic'

When `--type` IS specified:
- `--type code`: Only query with Jina, filter for jina modelId
- `--type text`: Only query with Nomic (with "search_query: " prefix), filter for nomic modelId
- `--type image`: Skip (no text-to-image search in Phase 5; print message to stderr: "Image search requires image query input (not yet supported)")

**Over-fetch strategy:**
- col-768 holds BOTH code and text vectors
- When querying for code: fetch `topK * 5` from col-768, filter by modelId to get code-only, then take topK
- When querying for text: fetch `topK * 5` from col-768, filter by modelId to get text-only, then take topK
- The `* 5` accounts for mixed vectors in the same collection (research recommendation)

**Top-K distribution:**
- `--top-k N` applies PER TYPE (10 code + 10 text by default = up to 20 total results)
- This maximizes context for AI coding assistants (the primary consumer)

**Chunk collapsing:**
- Apply existing collapsing logic (consecutive chunks from same file) per type independently
- Text chunks don't have line numbers (lineStart/lineEnd = 0), so text results show file path + score only

**JSON output format (grouped envelope):**
```json
{
  "query": "...",
  "code": [{ "file": "...", "lines": { "start": N, "end": N }, "score": N, "text": "..." }],
  "text": [{ "file": "...", "score": N, "text": "..." }],
  "totalIndexed": N,
  "searchScope": "."
}
```
- Only include `code` key if code results exist
- Only include `text` key if text results exist
- Text results omit `lines` field (no line tracking for text docs)

**Text output format:**
```
## Code

File: src/auth.ts | Lines: 10-25 | Relevance: 0.8542
    <chunk text indented>

## Text

File: docs/auth.md | Relevance: 0.7891
    <chunk text indented>
```
- Only print group headers that have results
- Text results omit "Lines:" part

**Pipeline lifecycle:**
- Load Jina pipeline, query, dispose
- Load Nomic pipeline, query, dispose
- Do NOT load both simultaneously (memory conservation)

**Error handling:**
- If col-768 query throws (empty collection), treat as zero results for that type
- If a model fails to load, skip that type and log to stderr
